# Kiro로 GeekNews Daily Pipeline 만들기

이 문서는 Kiro IDE를 사용하여 GeekNews Daily Pipeline을 처음부터 구현한 과정을 기록합니다.
어떤 프롬프트를 사용했고, Kiro의 어떤 기능이 활용되었는지를 중심으로 작성했습니다.

## 사용한 Kiro 기능

### 1. Spec (요구사항 → 설계 → 태스크)

Kiro의 Spec 기능을 사용하면 자연어로 요구사항을 정의하고, 이를 기반으로 설계 문서와 구현 태스크를 자동 생성할 수 있습니다.

프로젝트의 `.kiro/specs/geeknews-daily-pipeline/` 디렉토리에 3개의 문서가 생성되었습니다:

- `requirements.md` — 8개의 요구사항과 인수 조건
- `design.md` — 아키텍처, 컴포넌트 인터페이스, 데이터 모델, 정확성 속성(Property), 테스트 전략
- `tasks.md` — 12단계 구현 계획 (체크포인트 포함)

### 2. Steering

`.kiro/steering/geeknews-crawling-guide.md` 파일로 GeekNews HTML 구조에 대한 가이드를 제공했습니다.
Kiro는 이 steering 파일을 참고하여 크롤러의 CSS 선택자를 정확하게 구현했습니다.

### 3. Chat (대화형 개발)

Spec으로 기본 파이프라인을 구현한 뒤, Chat을 통해 AI 요약 기능을 추가했습니다.

## 프롬프트 기록

### Phase 1: 기본 파이프라인 구현 (Spec 사용)

Spec 생성 시 사용한 초기 프롬프트:

```
GeekNews(https://news.hada.io/new) 최신 글 페이지를 매일 크롤링하여 
IT 기사 정보를 수집하고, 로컬 Obsidian 저장소에 마크다운으로 정리한 뒤 
Slack으로 요약 알림을 보내는 자동화 파이프라인을 만들고 싶어.
AWS Lambda와 Python으로 구현하고, 로컬 테스트 및 배포 패키징을 지원해야 해.
```

Kiro가 이 프롬프트를 기반으로:
1. 8개의 요구사항(크롤링, 마크다운 생성, 저장, Slack 알림, 환경 변수, Lambda 핸들러, 배포, 스케줄링)을 도출
2. 각 요구사항에 대한 인수 조건(Acceptance Criteria)을 작성
3. 모듈별 설계와 인터페이스를 정의
4. 12개의 정확성 속성(Property)을 정의하고 속성 기반 테스트(Hypothesis) 전략을 수립
5. 체크포인트가 포함된 단계별 구현 태스크를 생성

이후 태스크를 하나씩 실행하면서 Kiro가 코드를 작성하고, 각 체크포인트에서 테스트를 실행하여 검증했습니다.

### Phase 2: AI 한국어 요약 기능 추가 (Chat 사용)

기본 파이프라인이 완성된 후, Chat에서 다음 프롬프트로 개선 방향을 논의했습니다:

```
현재는 저장한 md 파일과 slack으로 보내지는 문자가 한국인 사용자가 읽기 어렵게 되어 있어.
한국인 사용자에게 각 포스팅에서 가장 중요한 해당 내용을 잘 전달할 수 있는 방식은 무엇일까?
이를 통해 사용자는 인사이트를 얻을 수 있는 문서를 쉽게 접할 수 있어야 해.
```

Kiro가 현재 코드를 분석하고 LLM 기반 한국어 요약 단계 추가를 제안했습니다.
구체적인 마크다운/Slack 출력 예시와 함께 구현 방향을 보여줬습니다.

이어서:

```
이 방식을 반영하고 싶어. LLM API로는 github copilot을 활용하고 싶어. 
내가 학생 계정이 있거든.
```

이 프롬프트로 Kiro가:
1. GitHub Models API 문서를 웹에서 검색하여 엔드포인트와 인증 방식을 확인
2. `src/summarizer.py` 모듈을 새로 생성 (GitHub Models API 호출)
3. `Article` 데이터 모델에 `ai_summary`, `ai_insight` 필드 추가
4. 마크다운 생성기를 번호 매기기 + 핵심 요약 + 인사이트 섹션 형식으로 개선
5. Slack 알림을 기사별 블록 형태로 개선
6. Config에 `GITHUB_TOKEN`, `GITHUB_MODEL` 환경 변수 추가
7. 기존 테스트를 새 형식에 맞게 업데이트하고 전체 테스트 통과 확인

### Phase 3: 실행 및 검증

```
env 업데이트했어.
```

Kiro가 로컬에서 `python -m src.handler`를 실행하여 전체 파이프라인을 테스트했습니다.
Obsidian Vault 경로 문제(WSL vs Windows)를 자동으로 감지하고 수정한 뒤,
20건의 기사가 AI 요약과 함께 성공적으로 생성되는 것을 확인했습니다.

## Kiro 활용 팁

### Spec으로 시작하기

복잡한 프로젝트는 Chat 대신 Spec으로 시작하는 것이 좋습니다.
Spec은 요구사항 → 설계 → 태스크의 구조화된 흐름을 제공하고,
각 단계에서 사용자가 검토하고 피드백할 수 있습니다.

### Steering으로 도메인 지식 제공하기

크롤링 대상 사이트의 HTML 구조처럼 Kiro가 알 수 없는 도메인 지식은
steering 파일로 제공하면 정확한 구현이 가능합니다.

### Chat으로 점진적 개선하기

기본 구현이 완료된 후에는 Chat을 통해 기능을 추가하거나 개선할 수 있습니다.
Kiro는 기존 코드를 분석하고, 변경이 필요한 모든 파일을 자동으로 업데이트합니다.
테스트도 함께 수정해주므로 기존 기능이 깨지지 않는지 바로 확인할 수 있습니다.

### 환경 문제 자동 해결

로컬 실행 시 발생하는 경로 문제, 의존성 설치 등을 Kiro가 직접 실행하고 해결합니다.
에러가 발생하면 원인을 분석하고 수정한 뒤 다시 실행하는 과정을 자동으로 수행합니다.
